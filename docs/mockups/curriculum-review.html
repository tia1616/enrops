Behavior notes (for build)

  • Dora banner (top of right column). Replaces the duplicate input-card pattern. Counts fields where (confidence < 0.7) OR (extracted_value is null AND field is in Dora's flag list: age_range, class_size, format, session_types_supported, prerequisites, mid_term_skills, final_showcase). "Jump to first ↓" scrolls to the first flagged field. When zero flags remain, banner switches to the muted "All caught up" state.
  • Field-level "Dora flagged" badge. Each flagged field in Section B gets a soft gold-bordered chip in the label + a low-confidence ring on the input. Saving/editing the field clears the flag: sets human_approved=true + human_approved_by on the corresponding curriculum_extracted_fields row (or creates one for fields that didn't extract at all). The Dora banner count auto-decrements.
  • Dora is admin-portal only. No agent avatars or persona voice on parent / instructor / public registration surfaces.
  • Dora avatar. Placeholder is a plum circle with "D". Real art (Jessica has it) swaps in after first ship — just a CSS background-image change.
  • Section B fields. Pulled live from curricula + curriculum_extracted_fields. Schema additions (this chunk): class_size_min, class_size_max, prerequisites, mid_term_skills (text[]), final_showcase (text). Dropped: instructor_guide_notes already gone via commit b0bbce0. Already-existed addition: grade_min/grade_max shown via ages/grades toggle (whichever pair is populated).
  • Confidence indicators. Fields where confidence < 0.7 in curriculum_extracted_fields get a soft gold ring + tooltip "Double-check this one — we weren't fully sure." Never red.
  • Saving. Text inputs debounced 800ms after last keystroke. Chip add/remove + checkbox + toggle save immediately. Every save: (a) update the canonical column on curricula or curriculum_sessions, (b) for top-level fields, update curriculum_extracted_fields.human_edited_value + flip human_approved=true.
  • Sessions accordion. First session expanded by default. Session-level edits write only to curriculum_sessions (no per-session-field row in curriculum_extracted_fields — the whole sessions array is one row).
  • Session count is read-only. Changing it would require a re-extraction.
  • Publish flow (2-step modal). Step ①: confirm the curriculum name (editable inline) — it's the public-facing name. Step ②: fuzzy-match SQL query — normalized lowercase compare of curricula.name vs distinct programs.curriculum WHERE organization_id = current org AND curriculum_id IS NULL. If matches score high, show checkboxes per matched program run. On confirm: flip curricula.status='published' + set programs.curriculum_id = curricula.id on checked rows. Returns to /admin/curricula.
  • Save as draft. Status stays 'extracted'. Returns to library.
  • Org-context preload for the extraction prompt (follow-up commit, not this chunk). Edit supabase/functions/extract-curriculum-details/index.ts + prompts.ts to pull distinct age ranges, class-size ranges, and format from the caller's programs rows and inject as system-prompt context: "This org's existing programs typically use ages 6–10 or 7–12 (camps) with class sizes 4–20. Use this as a sanity check when extracting." Reduces hallucination on policy-shaped fields. Lands as small follow-up before next test upload.
  • Recap-template surface. Per-session recap_template is the parent portal entry after each class — not a per-session email. Default email cadence = mid-term + final only (mid-term uses curricula.mid_term_skills + sessions 1 to mid-point; final uses curricula.skills_overall). Provider-configurable email cadence (per-session / weekly / mid-term + final / final only / none) is deferred to a later chunk — out of scope here.
  • Parent-portal per-session view. Reads curriculum_sessions.{title, description, skills_practiced, parent_engagement_question, recap_template} — all already extracted.
  • Multi-tenant. Every query filters by organization_id from outlet context. No J2S hardcoding. Dora's persona copy is generic.
  • RLS. Curricula + sessions + extracted_fields already RLS-scoped per org. Updates use anon client + user JWT.
